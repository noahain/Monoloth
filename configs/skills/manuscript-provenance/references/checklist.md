# Manuscript Provenance Checklist

**Scope:** Verify that every data-carrying element in the manuscript traces to
code, scripts, or config — not manual entry.

**Severity markers:**

- ★★ CRITICAL — Core result with no provenance. Paper's claims unverifiable.
- ★ HIGH — Reproducibility gap in results or methodology.
- ◆ MEDIUM — Consistency/maintenance risk.
- ○ LOW — Minor provenance gap, cosmetic.

---

## §1. Value Provenance — Inline Numbers

Every number in the manuscript text (metrics, percentages, counts, p-values,
confidence intervals, thresholds, sizes, dates, version numbers) must trace
to a script output.

- [ ] ★★ **Abstract metrics**: Every quantitative claim in the abstract traces
      to a specific script output. These are the first numbers a reader sees.
- [ ] ★★ **Headline results**: Key findings reported in results section trace
      to script outputs with matching precision (decimal places, rounding).
- [ ] ★ **Methodology parameters**: Hyperparameters, architecture sizes, learning
      rates, batch sizes reported in methodology trace to config files or script
      arguments.
- [ ] ★ **Baseline comparisons**: Numbers attributed to baselines trace to
      baseline evaluation scripts or cited source with exact reproduction.
- [ ] ★ **Statistical measures**: p-values, confidence intervals, effect sizes
      trace to statistical test scripts.
- [ ] ◆ **Dataset statistics**: Counts, sizes, splits, class distributions trace
      to data processing scripts.
- [ ] ◆ **Computational cost**: Runtime, memory, hardware specs trace to
      benchmarking scripts or system info capture.
- [ ] ○ **Illustrative values**: Example values used for explanation (not results)
      — lower priority but still flagged if untraced.

**Detection method:** Extract all numeric tokens from `.tex` source. For each,
search script outputs (logs, CSVs, generated `.tex` fragments, result JSONs).
Flag any number appearing in results/methodology sections with no match.

## §2. Table Provenance

Tables are high-density data artifacts. Every cell, header, and ordering
decision must be computationally grounded.

- [ ] ★★ **Results tables**: Main results tables (typically Tables 1-3) are
      fully generated by scripts. No manual cell entry.
- [ ] ★ **Cell value matching**: Every numeric cell in generated tables matches
      the script output exactly (precision, rounding, formatting).
- [ ] ★ **Row ordering**: Row order determined by code (descending performance,
      alphabetical, grouped by category) — not manually arranged.
- [ ] ★ **Column ordering**: Column order determined by code or config — not
      manually arranged.
- [ ] ◆ **Header labels**: Column and row headers match code-defined names or
      have explicit display-name mappings in code.
- [ ] ◆ **Formatting rules**: Bold/italic for best values, significant figures,
      ± notation applied by generation script — not manual LaTeX markup.
- [ ] ◆ **Ablation tables**: Ablation study tables generated by the same pipeline
      as main results.
- [ ] ○ **Supplementary tables**: Tables in appendix/supplementary material also
      generated, not manually transcribed.
- [ ] ○ **Table notes**: Footnotes and annotations in tables match code output
      or are clearly editorial (not data-carrying).

**Detection method:** For each `tabular`/`table` environment, check if it is
`\input`-ed from a generated directory or defined inline. Inline tables with
numeric content are suspect. Compare cell values against script outputs.

---

## §3. Figure Provenance

Figures encode data visually. The generation chain must be unbroken from data
to final image file.

- [ ] ★★ **Main result figures**: Primary figures (plots, charts, diagrams
      showing results) produced by scripts that read result data.
- [ ] ★ **Script→file traceability**: Every `\includegraphics{path}` path
      matches an output path in a generation script.
- [ ] ★ **Deterministic generation**: Plotting scripts use fixed random seeds
      where randomness is involved (e.g., t-SNE, UMAP, stochastic layouts).
- [ ] ★ **Data source traceability**: Plotting scripts read from result files
      produced by analysis scripts — not from hardcoded arrays in the plotting
      script itself.
- [ ] ◆ **No post-editing**: Generated figures not modified in external tools
      (Illustrator, PowerPoint, GIMP). Check file metadata for editing software
      signatures.
- [ ] ◆ **Caption-value provenance**: Quantitative claims in captions (e.g.,
      "achieving 94.2%") trace to script outputs. _(Caption readability and
      self-containment owned by manuscript-review §12.)_
- [ ] ◆ **Axis ranges**: Axis limits set by data range or code config — not
      manually tuned to hide inconvenient data points.
- [ ] ◆ **Color/style config**: Color mappings, line styles, marker types
      defined in code config — not ad-hoc per script. _(Whether the resulting
      visuals are consistent across figures is owned by manuscript-review §12.)_
- [ ] ○ **Schematic/conceptual figures**: Architecture diagrams, flowcharts —
      lower priority since they are inherently manual, but flag if they contain
      data values that should trace to code.
- [ ] ○ **Figure format generation**: Script generates the output in the correct
      format directly — not manually exported from a GUI tool. _(Format recommendations
      owned by manuscript-review §12.)_

**Detection method:** List all `\includegraphics` paths. For each, search
codebase for scripts that write to that path (`savefig`, `plt.save`,
`write_image`, file write operations). Flag missing matches.

## §4. LaTeX Macro Audit

LaTeX macros are the primary mechanism for injecting computed values into
manuscript text. Every data-carrying macro must be script-generated.

- [ ] ★★ **No magic numbers in macros**: `\newcommand{\accuracy}{94.2}` defined
      directly in a `.tex` file = provenance failure. Must come from
      `\input{generated/metrics.tex}`.
- [ ] ★ **Generated macro files**: A script produces `.tex` files containing
      macro definitions, which the manuscript `\input`s.
- [ ] ★ **Macro-value consistency**: The value in the generated macro file matches
      the current script output (not a stale cached version).
- [ ] ★ **Complete macro coverage**: Every numeric value in running text is a
      macro reference, not a bare number. Pattern: `\accuracy` not `94.2`.
- [ ] ◆ **Macro naming convention**: Macro names reflect their semantic meaning
      (`\bestModelAccuracy`, not `\valA`).
- [ ] ◆ **Macro organization**: Related macros grouped in the same generated file
      (all metrics in `metrics.tex`, all dataset stats in `data-stats.tex`).
- [ ] ◆ **No unused macros**: Every defined macro is referenced in the manuscript.
      Orphaned macros suggest stale or removed content.
- [ ] ○ **Formatting macros**: Number formatting (percentage signs, ± symbols,
      decimal places) applied in the macro definition or via a formatting command —
      not inline in text.
- [ ] ★ **Bare number detection**: Every numeric value in running text that
      represents a result, parameter, or computed quantity and is NOT a macro
      reference. These are values that SHOULD be macros but aren't — they have
      no provenance chain by definition. Pattern: `achieved 94.2\%` (bare) vs.
      `achieved \accuracy\%` (macro). Bare numbers in results and methodology
      sections are HIGH priority; in background/related work (citing others'
      numbers) are MEDIUM.
- [ ] ★ **Macro manifest generation**: Produce a structured manifest of all
      macro-injected values for consumption by manuscript-review Pass 13. Each
      entry: macro name, resolved value, source chain, manuscript location(s),
      provenance classification. Also include bare numbers that should be macros.
      Saved as `[manuscript-name]-macro-manifest.json`.

**Detection method:** Parse all `\newcommand`, `\def`, `\pgfmathsetmacro`,
`\providecommand` definitions. Classify by source file: if defined in a
`generated/` directory → TRACED. If defined in main `.tex` → check if value
is purely structural (formatting helper) or data-carrying (contains a number
or data-derived string). Additionally, scan all running text for bare numeric
tokens (digits, decimal points, percentage signs, ± symbols) not inside a
macro reference. Classify each as data-carrying or incidental (page numbers,
section references, dates).

## §5. Terminology Provenance

Named concepts in the manuscript (modes, mechanisms, strategies, method
variants, dataset names) must match code definitions.

- [ ] ★ **Mode/variant names**: Names of methods, models, configurations in the
      manuscript match code identifiers (enum values, class names, config keys).
- [ ] ★ **Consistent naming**: The same concept uses the same name throughout
      the manuscript AND in code. No "greedy search" in text when code calls it
      `beam_search_greedy`.
- [ ] ★ **Complete coverage**: Every named variant/mode in the manuscript exists
      in code. No "phantom methods" discussed but never implemented.
- [ ] ◆ **Display-name mapping**: If code uses `snake_case` identifiers and the
      manuscript uses "Title Case Display Names", an explicit mapping exists in
      code or config (not just the author's mental model).
- [ ] ◆ **Category consistency**: Categories, groups, or classes discussed in
      text match the groupings in code (e.g., dataset splits, error categories,
      feature groups).
- [ ] ◆ **Acronym-code alignment**: Acronyms introduced in the manuscript for
      methods/components align with code naming (e.g., if the paper calls it "MHA",
      the code module is `MultiHeadAttention` or `MHA`, not `attention_layer`).
      _(Abbreviation consistency within the document owned by manuscript-review §14.)_
- [ ] ○ **Deprecated terms**: No terms from earlier code versions that were
      renamed but the manuscript still uses the old name.

**Detection method:** Extract all capitalized multi-word terms, quoted terms,
and introduced names from manuscript. Search codebase for exact and fuzzy
matches. Flag terms with no code correspondence.

## §6. Ordering Provenance

The sequence in which items appear in tables, lists, and discussions is an
implicit assertion. It must be computationally determined.

- [ ] ★ **Results table row order**: Rows ordered by a code-determined criterion
      (descending accuracy, alphabetical, grouped by category from code).
- [ ] ★ **Method comparison order**: When methods are compared in text, the
      ordering matches code output (e.g., sorted by performance metric).
- [ ] ◆ **Feature/variable listing order**: When features, variables, or
      components are listed, the order comes from code (importance ranking, enum
      definition order, config file order).
- [ ] ◆ **Enumeration consistency**: If items are listed in order in one place,
      the same order appears everywhere in the manuscript.
- [ ] ◆ **Table column order**: Column arrangement follows a code-defined
      sequence (e.g., metrics ordered by importance, datasets ordered by size).
- [ ] ○ **Discussion order**: When results are discussed in prose, they follow
      the same ordering as the tables/figures they reference.

**Detection method:** For each ordered list/table, determine the ordering
criterion. Check if code output produces items in that order. Flag ordering
mismatches.

## §7. Pipeline Completeness

A complete pipeline means: one command, raw data in, PDF out. Partial
pipelines get partial credit.

- [ ] ★★ **Single entry point**: A `Makefile`, `snakemake`, `dvc.yaml`,
      `justfile`, `run.sh`, or equivalent exists that orchestrates the full
      pipeline.
- [ ] ★ **Data → results**: Scripts that process raw data into result files are
      part of the pipeline.
- [ ] ★ **Results → manuscript artifacts**: Scripts that convert results into
      tables, figures, and macro files are part of the pipeline.
- [ ] ★ **Artifacts → PDF**: LaTeX compilation is part of the pipeline (or at
      minimum documented).
- [ ] ★ **Dependency tracking**: The pipeline knows which outputs depend on which
      inputs. Changing a script re-runs only affected downstream steps.
- [ ] ◆ **Idempotency**: Running the pipeline twice produces identical output
      (fixed seeds, deterministic operations).
- [ ] ◆ **Incremental execution**: Pipeline supports partial re-execution without
      full restart.
- [ ] ◆ **Error handling in pipeline**: Pipeline fails loudly on script errors —
      does not silently skip steps or produce partial output.
- [ ] ○ **CI/CD integration**: Pipeline runs in CI (GitHub Actions, GitLab CI)
      to catch breakage automatically.
- [ ] ○ **Pipeline documentation**: Entry point and steps documented (even a
      comment block in the Makefile counts).

**Detection method:** Search for `Makefile`, `snakemake`, `dvc.yaml`,
`justfile`, `run.sh`, `run.py`, `pipeline.*` in project root. If found,
trace the dependency graph. If not found, flag as CRITICAL.

## §8. Code/Config Separation

Configuration values belong in config files. Code reads config; code does not
define config.

- [ ] ★ **Hyperparameters in config**: Learning rates, batch sizes, epochs,
      architecture parameters defined in config files — not hardcoded in scripts.
- [ ] ★ **Model names in config**: Model identifiers, checkpoint paths, pretrained
      model references in config — not hardcoded.
- [ ] ★ **Thresholds in config**: Decision thresholds, cutoff values, filtering
      criteria in config — not hardcoded.
- [ ] ◆ **File paths relative**: All paths in code and config are relative to
      project root — no absolute paths.
- [ ] ◆ **Single config entry point**: One config file (or a clearly structured
      set) — not settings scattered across scripts.
- [ ] ◆ **Config format consistency**: All config in the same format (TOML, YAML,
      JSON) — not mixed formats.
- [ ] ◆ **No secrets in code**: API keys, credentials, machine-specific paths
      absent from committed code. In `.env` or environment variables.
- [ ] ○ **Config documentation**: Config keys documented (comments in config file,
      or separate schema).
- [ ] ○ **Default value handling**: Scripts fail on missing config keys rather
      than silently using defaults.

**Detection method:** Search scripts for hardcoded numeric literals, string
literals that look like paths/model names, and assignment patterns that should
be config-driven. Cross-reference with config files.

## §9. Stale Output Detection

Outputs must be current relative to their generating scripts and input data.

- [ ] ★ **Script-output freshness**: Every output file has a modification
      timestamp equal to or newer than its generating script.
- [ ] ★ **Data-output freshness**: Every output file has a modification timestamp
      equal to or newer than the input data it reads.
- [ ] ◆ **Orphaned outputs**: Output files with no corresponding generating
      script (leftover from deleted/renamed scripts).
- [ ] ◆ **Unrun scripts**: Scripts with no corresponding output files (new or
      modified scripts that haven't been executed).
- [ ] ◆ **Generated LaTeX freshness**: `generated/*.tex` macro files current
      relative to the scripts that produce them.
- [ ] ○ **Figure freshness**: Figure files current relative to both plotting
      scripts and the data they plot.
- [ ] ○ **Intermediate file freshness**: Cached intermediate results current
      relative to upstream scripts.

**Detection method:** Build a dependency graph from script I/O analysis.
Compare modification timestamps along each edge. Flag reverse-chronological
edges (output older than source).

## §10. Version Pinning & Environment Reproducibility

The computational environment must be reproducible on another machine.

- [ ] ★ **Dependency lock files**: `requirements.txt` with pinned versions,
      `poetry.lock`, `conda-lock.yml`, `uv.lock`, `Pipfile.lock` — at least one
      exists.
- [ ] ★ **Language version specified**: Python version, R version, Julia version
      specified (`.python-version`, `runtime.txt`, conda env).
- [ ] ◆ **Data versioning**: Input data tracked via DVC, git-lfs, checksums, or
      versioned URLs — not untracked local files.
- [ ] ◆ **Manuscript-code co-versioning**: Manuscript and code in the same
      repository or linked via git submodules/tags.
- [ ] ◆ **Container definition**: `Dockerfile`, `singularity.def`, or `nix`
      config for full environment reproduction.
- [ ] ○ **OS/hardware dependencies documented**: CUDA version, GPU requirements,
      platform-specific notes.
- [ ] ○ **Seed documentation**: Random seeds listed and their scope documented
      (which operations they control).

**Detection method:** Search for lock files, version specifications, container
definitions, and data versioning configs in project root and standard locations.

---

## §11. Cross-Reference Integrity

Internal cross-references between manuscript sections and code artifacts must
be consistent.

- [ ] ★ **Script references in manuscript**: If the manuscript mentions "Script X"
      or "see code", the referenced file/function exists.
- [ ] ★ **Appendix code listings**: Code snippets in appendix match current
      source code — not stale copies.
- [ ] ◆ **Repository URL validity**: If a code repository URL is given, it
      resolves and contains the expected content.
- [ ] ◆ **README accuracy**: Repository README describes the actual pipeline
      steps, not an outdated version.
- [ ] ○ **Line number references**: If specific line numbers are cited, they
      match current file state.

**Detection method:** Extract all references to code/scripts/repos from
manuscript. Verify each against the actual codebase.

---

## §12. Manuscript-Review Integration Points

Provenance findings that should be fed back to manuscript-review.

- [ ] ◆ **§6 (Methodology) feedback**: UNTRACED methodology parameters →
      manuscript-review reproducibility sufficiency failure.
- [ ] ◆ **§7 (Results) feedback**: UNTRACED result values → manuscript-review
      result-claim alignment concern.
- [ ] ◆ **§12 (Figures/Tables) feedback**: MANUAL figures/tables → manuscript-review
      quality concern.
- [ ] ◆ **§14 (Nomenclature) feedback**: Terminology mismatches → manuscript-review
      abbreviation/naming concern.
- [ ] ◆ **§17 (Compliance) feedback**: Missing code/data availability → manuscript-review
      compliance concern.

**Detection method:** If a manuscript-review report exists, load it and
cross-reference. If not, note integration recommendations in the provenance
report.

---

## Priority Legend

- **★★ CRITICAL** — Core result unverifiable from code. Paper's central claims
  lack computational grounding.
- **★ HIGH** — Reproducibility gap. A competent reader cannot regenerate this
  artifact from the codebase.
- **◆ MEDIUM** — Consistency or maintenance risk. Not blocking but degrades
  confidence in reproducibility.
- **○ LOW** — Minor gap. Cosmetic or low-impact provenance issue.

## Audit Focus Order

1. Abstract and headline metrics (§1 ★★ items)
2. Main results tables (§2 ★★ items)
3. Main result figures (§3 ★★ items)
4. Pipeline existence (§7 ★★ items)
5. LaTeX macro hygiene (§4 ★★ items)
6. Remaining HIGH items across all sections
7. MEDIUM items
8. LOW items
